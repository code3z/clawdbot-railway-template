## Lesson Reinforced: Check Data Before Theorizing

When P&L doesn't match expectations, first query:
1. `SELECT created_at, realized_pnl, total_trades FROM portfolios WHERE model=?`
2. Compare total_trades to actual settled count in trades table
3. Compute true P&L directly: `SELECT sum(pnl) FROM trades WHERE model=? AND status IN ('won','lost','exited_early')`

### nws_forecast_trader — `less` Market Bug (Signal + Settler)

**Signal bug**: `else: # greater` branch handled both `greater` AND `less` strike types identically. For `less` markets (YES = val < threshold), `nws_val > threshold → YES` is backwards — should be NO. Fixed: added explicit `elif strike_type == "less":` branch with inverted logic.

**Settler bug**: `yes_won = cli_val > trade.threshold` used for all non-between markets. For `less` markets, correct logic is `cli_val < threshold`. Fixed: added `elif trade.strike_type == "less": yes_won = cli_val < trade.threshold` before the else.

**Why it wasn't obvious**: `strike_type` was never stored in `Trade.create()` (always NULL). Settler's `between` check (`trade.strike_type == "between"`) always failed → fell through to the wrong `else` clause. All other traders store strike_type correctly. Only `nws_forecast_trader.py` was affected.
