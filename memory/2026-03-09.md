### Key feature insights
- `delta_next_c` (+0.26 corr): strongest predictor — if next 5-min reading is in higher bucket, this reading is hi_f
- `delta_prev_c` (-0.21 corr): if prev was colder (rising into bucket), reading is hi_f
- `next_bucket_delta` (+0.21): if T-group after is warmer, this reading is hi_f
- `prev_bucket_delta` (+0.18): if prev T-group was warmer (temp dropped here), reading is lo_f
- `local_hour`, `day_of_year`: ~zero correlation (rounding ambiguity is uniform across time)

### Architecture note for integration
The expanded model needs different features than the existing 11/16-feature models.
The `version` key in the pkl payload is `"expanded_v1"` — can use that to dispatch.
The 8 new features come from the Synoptic 5-min obs sequence (wethr push stream data),
not from METAR T-group (which the old 16-feature model used).

At inference, `obs_seq` contains wethr push readings with `precision_level=0` (5-min HF-METAR)
and `precision_level=1` (hourly METAR). The expanded model uses only `precision_level=0` readings.

Feature extraction from obs_seq for the expanded model:
- Find peak_c readings in obs_seq (precision_level=0, temp_c == peak_c)
- T-group = consecutive readings at same c_bucket around the peak reading
- prev/next bucket delta = c_bucket of adjacent T-groups
- delta_prev_c = temp_c[peak_reading - 1] - peak_c
- delta_next_c = temp_c[peak_reading + 1] - peak_c  
- local_hour from peak_obs["obs_time"] + LST offset
- day_of_year from same

## Exploration: v2 features + ceiling analysis

### What we tried
1. **v2 (13 features)**: added c_bucket, delta_30min_c, delta_60min_c, tgroup_total, tgroup_position
   - Result: Test Brier 0.1713 vs v1's 0.1722 — only **+0.5% improvement**
   - `delta_30min_c` had -0.210 correlation but was redundant once model learned interactions

2. **Solar/time features**: hours_to_sunset, is_pre_noon, is_post_peak_window
   - All had |corr| < 0.015 with label. Label mean by hour: 0.46–0.55 (essentially flat)
   - Time of day adds zero signal once trajectory direction (delta_prev/next_c) is known

3. **Hourly METAR precision (rejected)**: IEM hourly stores whole °F integers (converted from whole °C METAR)
   - Same bucket ambiguity as 5-min data. Doesn't help for disambiguation.

### Conclusion
**Practical ceiling for 5-min ASOS data is ~0.172 Brier / AUC 0.82.**
The ceiling is the measurement resolution: ASOS reports whole °C (~1.8°F bucket).
Resolving within 1°F requires finer sensors or higher-frequency data.

**Production model: phoenix_expanded.pkl (v1, 8 features)**
Simpler, better ratio (1.07x vs 1.09x), essentially identical to v2 test performance.

To beat 0.172 would require: sub-°C sensor precision OR 1-min sampling rate.

### twc_morning_trader: top-up mode added (commit 56201a6)
New `top-up` command and `com.trady.twc-topup-trader` launchctl agent (fires 1 PM ET daily, 18:00 UTC).

**Flow:**
1. Fetch latest TWC for all cities
2. Audit all resting orders — cancel any whose direction is no longer supported by TWC (margin < 2°F or direction flipped)
3. Scan candidates with `top_up=True` mode:
   - **New tickers**: added normally
   - **Increase**: resting order exists but `committed < MAX_DOLLARS` → add up to remaining capacity
   - **Double-down**: filled trade today, TWC still valid → buy more (additional separate order, not cancel+repost)

- `audit_resting_orders()`: cancels stale orders, marks them `void` in DB
- `scan_and_build_candidates(top_up=True)`: skips dedup for re-entry tickers, reduces `dollar_size` by already-committed amount
- For increasing existing orders: places a second separate order (doesn't cancel+repost — avoids queue position risk)

## Session: CLI Trader Push Debugging + DSM/6-hr METAR Research (late evening 2026-03-09)

### CLI push trigger seq-reset bug — FIXED (commit 8d0cd51)

**Root cause:** wethr push client keeps its sequence counter (`_trigger_seq`) in memory. When the push client restarts, the counter resets to 0 and counts up from 1, 2, 3... The cli_push_trigger daemon saves `last_seq` to disk (state file). After a push client restart, the daemon still had `last_seq=704` while new events started at seq=393. The daemon's check `seq <= last_seq` (393 ≤ 704) caused every event to be silently skipped with "skip" log.

**Symptom:** Every city timed out (5-min timeout) or was silently skipped. No CLI trades fired the entire day despite wethr receiving CLI events.

**Fix applied:**
1. Manually reset state file `last_seq` to 392 (below next event seq=393)
2. Added seq-reset detection to daemon: if `last_seq - seq > 50`, assume push client restarted and reset `last_seq` to `seq - 1`

**Verification:** LAX fired at 9:39 PM and completed in 12 seconds (live + paper both exit=0). Working correctly.

**This will happen again** if push client restarts and the gap between old last_seq and new seq is > 50. The detection threshold of 50 handles normal drift.

### Three-part gate removed then restored (commit 8f5e6b5)

**Why it was removed:** In commit `803329a` we added the `--confirmed-high` push bypass to skip the 90-min NWS polling loop. The gate (`check_confidence()`) was tightly coupled to the polling infrastructure (needed valid_hour, peak_local_hour, issued_dt). When bypass was added, the gate was stripped too.

**Why that was wrong:** CLI firing at 1 PM with OM-predicted peak at 4 PM — no gate means we'd trade it. The push event carries `high_f` but NOT `valid_hour` or `peak_local_hour`.

**Fix:** In push mode, do a SINGLE one-shot NWS CLI fetch (~1s) to get valid_hour and peak_local_hour, then run the full three-part gate. No polling loop. If fetch fails → gate rejects (conservative). Gate failures logged as `gate_reject` events.

### DSM vs CLI empirical check

Compared DSM events from push log against CLI events from same log for 62 matching station-dates:
- **49/62 exact (79%)**, 5/62 off by ±1°F (8%), 8/62 larger (all date-attribution artifacts)
- The large discrepancies (13°F, 14°F, 23°F) are CLI events from midnight/1 AM reports that cover the PREVIOUS day, mislabeled with the issuance date. Ian corrected this: after-midnight CLIs are for the previous day (it's in MEMORY.md).
- True measurement discrepancy: ±1°F on 8% of cases. DSM consistently runs +1°F above CLI when they differ.
- **Conclusion:** DSM is essentially same measurement methodology as CLI. For trading, apply 1°F conservative buffer: only trade `greater YES` if DSM high > threshold + 1.

### 6-hour METAR max empirical check

Fetched from Synoptic API (`air_temp_high_6_hour_set_1`). Synoptic timeseries returns 0.1°C precision → single exact °F conversion (no bucket ambiguity — cleaner than 5-min HF-METAR).

Compared 26 station-dates (PHX, MIA, DFW, BOS, LAS):
- **22/26 exact (85%)**, 1/26 off by ±1°F, 3/26 off by -2°F
- The 6-hr max is ALWAYS ≤ CLI (never overestimates). It's a confirmed lower bound on the daily high.
- The -2°F cases: true daily peak occurred outside the 6-hr window sampled. The gate would catch these — if OM predicts peak at 3 PM, we'd use the 18Z-00Z window (1-7 PM), not 12Z-18Z (7 AM-1 PM).
- NWS API returns only `maxTemperatureLast24Hours` (not 6-hr). Must use Synoptic for `air_temp_high_6_hour_set_1`.
