### What happened
At 04:10 UTC heartbeat, discovered two sets of trading processes running simultaneously:
- **Old set (PIDs ~1038xxx)**: spawned by a previous/dead orchestrator, with `-W ignore::UserWarning:sklearn` in cmdline

### Remaining issue for Ian
**Orchestrator crash-loop for dashboard is STILL ongoing** (even after fix). The orchestrator is spawning new dashboard instances every ~6s that fail with "Address already in use" because 2627649 holds the port. This means the orchestrator has a bug: it doesn't detect that a previously-spawned instance is already running.

**This is a bug in orchestrator.py**: the keep_alive logic doesn't handle the case where a child process exits with error but ANOTHER instance of the same script is already running (from a previous failed spawn that actually succeeded). The orchestrator keeps spawning new ones in a crash-restart loop.

**Immediate risk**: None — trading daemons are fine, dashboard works. But the orchestrator is burning CPU/logs on the crash loop.

**Fix needed**: orchestrator.py should check if a port is already bound / process already alive before spawning. Or it needs a proper PID tracking mechanism.
