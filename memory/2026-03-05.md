## Wethr 409 Incident — Server-Side Session Timeout Much Longer Than Expected

### What happened
- Multiple `launchctl kickstart -k` calls during refactor review caused the push client to grab/release connections rapidly
- Wethr server held the "active connection" slot for ~20 minutes after our process died (TCP FIN sent, but server uses application-level session tracking — doesn't notice until it writes to the dead socket)
- Push client was stuck in 409 loop from ~7:47 PM to 8:03 PM ET (~16 minutes)
- PT trading window opened during this outage (7 PM ET)

### Fix
1. Reduced 409 retry from 300s → 30s in `wethr_push_client.py` (catches reconnect ASAP when server clears)
2. Kill-9'd the stuck instance; launchctl restarted with new code → connected at 8:03 PM ET
3. Manually launched PT traders (`--tz PT` live + paper, `-u` unbuffered) to cover the window

### Lesson
- Never do multiple `launchctl kickstart -k` calls in quick succession on the push client — each attempt resets the server's timeout clock
- Wethr server session timeout ≈ 20 min (not 5 min as implied). 30s retry is correct.
- `kill -9 <PID>` followed by single `launchctl kickstart` is the right restart sequence

## Wethr 409 — Normal vs Abnormal (clarification)

- **Normal operation**: connection runs for hours, drops occasionally (server-side), reconnects in seconds via `last_event_id` replay. No 409.

- March 3 reconnect cluster (many in ~10 min) = wethr server instability, not our side.
- **Rule: never restart the push client more than once in a 20-min window.** If restart needed: kill once, wait for launchctl to bring it back, don't touch it again.

### NBM GRIB2 Exploration (AWS S3 open data)
- NBM GRIB2 files on `noaa-nbm-grib2-pds.s3.amazonaws.com` — free, no auth
- Structure: `blend.YYYYMMDD/HH/core/blend.tHHz.core.fFFF.co.grib2`
- Also: `blend.YYYYMMDD/HH/text/` (NBS text files) and `blend.YYYYMMDD/HH/qmd/` (precipitation percentiles only, no temp)
- TMAX variable in GRIB2: appears at f036 (stepRange 24-36), f045 (stepRange 33-45), etc. — only every 12h, only starting ~24h out
- **GRIB has decimal precision** (e.g., 83.696°F) vs NBS text integer rounding

### GRIB Nearest-Neighbor Problem
- Simple `argmin(distance)` grid-point lookup fails for stations near grid cell edges
- Chicago KMDW: nearest cell = 39.49°F, adjacent cell = 65.41°F (lake/land boundary artifact)
- Houston: similar issue — nearest cell gave 79°F, adjacent gave 83°F (matching NBS)
- NBS text uses station ICAO lookup, not spatial interpolation → inherently more reliable

### NBM Has No Decimal Precision — Confirmed
- All NBM text products (nbstx, nbhtx, nbptx, nbxtx) are integer °F
- NWS API "decimals" are just integer °F converted to Celsius (28.3333°C = exactly 83°F)
- NOAA/NWS works in integer °F natively by design, not as a rounding artifact
- **Conclusion**: NBM's unique value is σ (XND field), not point forecast precision
- For decimal station forecasts: use VC, Tomorrow.io, OM models
