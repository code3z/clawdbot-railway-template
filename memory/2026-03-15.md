### bidb smart raise/drop guards (commit c70a962)
- Replaced blunt "never up" fix (08ce061) with nuanced raise/drop rules
- **Raise logic** (upward reprice from bidb):
  - If edge at new price > 20%: only raise if `_qty_above > 2` (few contracts above us → market may come down to us)
  - If edge at new price 8-20%: only raise if `_qty_above > 30` (need thick depth to justify chasing)
  - Rationale: deep depth above us = market genuinely not coming down; raise to stay relevant. Thin depth = be patient.
- **Drop logic** (downward reprice from bidb):
  - Only drop if `_second_bid_below < current_limit - 0.01` (no one within 1¢ below us)

**Market close timing:**
- Markets close at **midnight LOCAL CLOCK TIME** (not LST)

**Settlement uses LST (Local Standard Time):**

**Key orderbook API facts:**
- Correct endpoint: `GET /markets/{ticker}/orderbook` → `orderbook_fp` (NOT `orderbook`)
- Fields: `no_dollars` (NO bids), `yes_dollars` (YES bids), sorted ASCENDING by price
- Best bid = LAST element (highest price)
- `yes_ask = 1 - float(no_bids[-1][0])` (implied YES ask from best NO bid)
- `no_ask = 1 - float(yes_bids[-1][0])` (implied NO ask from best YES bid)
- Markets API `yes_ask`/`no_ask` fields are unreliable — always use orderbook endpoint directly

### End-of-day trader design

**Full spec written:** `polymarket/specs/end_of_day_trader_spec.md`

**Two signals:**

**Signal 1 — Conservative floor (NO trades):**
- `projected_minimum = min(running_min_f, adjusted_twc_min_lst)`
- `adjusted_twc_min_lst = twc_15min_min_lst + (current_obs_lower_f - twc_current_f)`
- For BETWEEN: bet NO "above range" if projected_min > cap; NO "below range" if running_min < floor
- For LESS (YES = low ≤ cap): bet NO if projected_min > cap
- For GREATER (YES = low ≥ floor): bet NO if running_min < floor (already confirmed below)

**Signal 2 — Confirmed bucket / retail bias (YES on BETWEEN):**
- Prerequisite: projected_minimum ≥ running_min (temp locked = daily min won't go lower)
- Confirmation: preliminary CLI (released ~5pm local) OR 6-hr NWS report
  - CLI must have been issued AFTER the last observation of the lowest C reading
  - 6-hr report must span the ENTIRE time window where the lowest C was observed
- Signal 2 resolves C→F ambiguity (e.g., 13°C → 55 OR 56°F; CLI says 56 definitively)
- Bet YES if confirmed_low_f is in [floor, cap]

**Signal 2 depends on Signal 1:** both required for YES trades. Signal 1 alone is enough for NO trades.

**Parameters:**
- Paper only, $50/trade, 2% min edge
- Start scanning 5 hours before midnight LST per city
- LOW markets only (HIGH markets: future work)
- Taker first (hit asks), post limit bid if no ask available (rare)
- Dedup by (ticker, signal_type) per LST day

## Rule: Never set low timeouts on agent/cron jobs

Agent jobs (cron jobs, subagents doing real work) need real timeout headroom.
- **10 minutes (600s) minimum** for any agent cron job that runs scripts, queries DBs, or checks logs
- 120s or 180s is way too short — agent jobs do actual work, not just HTTP pings
- When creating or editing a cron job with `--timeout-seconds`, default to 600 unless it's genuinely a trivial one-liner
- The morning health check timed out at 120s because it was set without thinking about actual agent execution time
