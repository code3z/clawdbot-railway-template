## TWC Trough Lookup Table — Research (~12 AM ET)

### Context
Ian asked for a simpler alternative to the existing `trough_passed_signal` (scikit logistic model).
Current production model uses: `fc_gap`, `local_hour`, `consec_rising` → P(trough passed).
New idea: simpler lookup table keyed on `twc_remaining_downside = running_min_f - twc_low_f`.
Ian's hypothesis: TWC is more accurate for lows, so -1°F on TWC ≈ -2 to -3°F on OM.

### TWC vs OM accuracy (limited data — 6 TWC vs 80 OM matched city-dates)
- TWC low MAE: 1.50°F, bias: -1.17°F (TWC forecasts ~1°F too cold)
- OM low MAE: 5.03°F, bias: +0.25°F
- Ratio: 0.30x → -1°F on TWC ≈ -3.4°F on OM
- Hypothesis confirmed directionally, but only 6 TWC data points — very noisy

### Training/inference mismatch — key design decision
- IEM 1-min `tmpf` is fine for training (original whole-°F, not derived from °C)
- IEM has gaps (NCEI manual collection), especially in afternoons — noisy running_min
- At inference, `min_lower_f` comes from °C bucket lower bound (push obs), not exact °F
- Fix: at training time, convert IEM °F → °C bucket → lo_f/hi_f before computing feature
- Build TWO tables: LO table (uses lo_f = conservative) and HI table (uses hi_f = optimistic)
- At inference: look up both and the true probability is between them

### Lookup tables (102 city-dates, Feb 23 – Mar 2, ~7 unique dates — NOISY)
Build script: `polymarket/intraday/build_twc_trough_table.py`

**LO table (min_lower_f conservative):**
- `-3_or_less`: P=1.000 (N=134) ← trade
- `-2`:         P=0.793 (N=87)  ← borderline
- `-1`:         P=0.979 (N=143) ← trade (anomaly — small sample)
- `0`:          P=0.933 (N=105) ← anomaly (likely seasonal bias)

**HI table (min_upper_f optimistic):**
- `-3_or_less`: P=1.000 (N=102) ← trade
- `-2`:         P=1.000 (N=41)  ← trade
- `-1`:         P=0.816 (N=98)  ← borderline
- `0`:          P=0.979 (N=144) ← anomaly

### Practical inference rule (pending more data)
- If `lo_f - twc_low ≤ -3`: trade (both tables agree: 100%)
- If `hi_f - twc_low ≤ -2` AND `lo_f - twc_low ≥ -3`: probably trade (HI=100%, LO=79%)
- Both tables: look up bucket in LO table using lo_f, look up in HI table using hi_f, report both P values

### Next steps
- Wait 4-6 more weeks for enough TWC data to build a reliable table
- The 0/1 bucket anomalies (95%+ P when running_min still at/above TWC forecast) are small-sample noise from winter/warm-climate cities where the low hits at 6 AM and only warms from there
- Do NOT integrate into production until table is monotonic and sample is larger
- A/B test vs existing scikit `trough_passed_signal` when ready

## trough_passed_v2_signal — A/B Test Wired (~1 AM ET)

### What was built
Ian asked: "if lo_f is already 1°F below TWC's estimate, what's the prob it goes even lower?"
Answer: TWC forecasts too cold (-1.2°F cold bias), so if you're already 1°F below TWC, you're
likely 2–3°F below OM's implied forecast. P(further drop) is low.

New signal `trough_passed_v2_signal` — three-part gate:
1. `min_lower_f <= twc_low - 1` (conservative obs bound already 1°F below TWC forecast)
2. `local_hour > om_predicted_trough_hour` (past OM's predicted trough time)
3. `consec_rising >= 1` (at least one rising obs since minimum)
Gate confidence fixed at 0.87 (conservative; calibrate as data accumulates).

### Architecture
- New paper model: `nws_obs_trough_twc` (added to `ALL_MODELS` in `db.py`)
- Helper function `_run_trough_v2_ab()` extracted to avoid duplication
- Both push path AND `scan_and_trade` path evaluate v2 independently
- Separate dedup set `_session_traded_paper_v2` — won't block main signal on same ticker
- New caches: `_OM_TROUGH_HOUR_CACHE`, `_TWC_LOW_CACHE` (keyed by city+date_str)
- TWC low source: earliest matching entry in `forecast_logs/forecasts.jsonl` per city+date

### Key A/B design point
v2 evaluates for ALL low markets before `if direction is None: continue` — so it fires
even when the scikit trough model doesn't, and vice versa. First version had this wrong
(v2 was placed after the continue, so it only ran when the main signal fired). Fixed.

### Portfolio separation note
`nws_obs_confirmed` = ALL obs trader signals (confirmed, peak_passed, trough, late_night etc.)
`nws_obs_trough_twc` = only v2 TWC gate trades
To compare just trough signals: filter `nws_obs_confirmed` by `tags LIKE '%trough_passed%'`.
