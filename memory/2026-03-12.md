## TWC DST Bug — Root Cause & Fix (commit 2a3e091, ~midnight March 12)

### The Bug
`twc_forecast.py` had `_dst_info()` returning `True` only on the ONE day clocks spring forward (March 8). The disputed-window correction (fetching hourly temps for midnight→1AM LDT) only fired on that single day. For the remaining ~230 days of DST, `is_dst_day=False` → correction dead → **every DST day, the NWS settlement window extends 1 hour past what TWC's calendarDay covers, and we miss it**.

### The Fix
Added `_is_dst_active(target_date, tz_name)` — checks `ZoneInfo(tz_name)` DST offset at noon of that day. Returns True for ALL DST-active days. Arizona/Hawaii automatically return False (no DST). Both `get_twc_forecast` and `fetch_twc_lst` now use `_is_dst_active` instead of `is_dst and dst_dir == +1`.

### Verified live (NYC March 12)
- `_is_dst_active("2026-03-12", "America/New_York")` → True ✅
- `_is_dst_active("2026-03-12", "America/Phoenix")` → False ✅ (no DST in AZ)
- NYC: cal_lo=54°F → lst_lo=37°F (disputed-window correction working)
- Denver, Chicago: cal_lo unchanged (no cold front in disputed window)

### calendarDayTemperatureMin — authoritative definition confirmed
From TWC API docs example: `temperatureMin=59, calendarDayTemperatureMin=65` for the same day. `calendarDayTemperatureMin` is **strictly midnight-to-midnight** local. `temperatureMin` is the night period (7 PM → 7 AM next morning), extends past midnight, therefore colder. Ian was right.

### Why cal_lo=54°F vs hourly min=37°F for NYC March 12
**Model staleness, not definition.** Daily forecast updates infrequently (set by most recent model run, probably yesterday evening). Hourly updates more frequently. The cold front arriving mid-morning March 12 was already in the hourly by midnight, but the daily cal_lo still reflected the older run. Both are midnight-to-midnight; just different model runs.

### Implication for our correction
The disputed-window fix does two things:
1. Extends the settlement window by 1 hour (midnight LDT → 1 AM LDT) — the main intent
2. Implicitly uses a fresher forecast source (hourly vs daily) — bonus accuracy

## TWC LST Window Refactor (commits 52ed0eb, c1d40e7, ~1:30 AM March 12)

### What changed
`fetch_twc_lst` and `get_twc_forecast` in `twc_forecast.py` now use the full LST-day
hourly window for temperature correction, not just the 1-hour disputed window.

**Old behavior:** only checked disputed 1 hour (midnight LDT D+1 → 1 AM LDT D+1).
Missed TWC daily model staleness for the rest of the day.

**New behavior:**
- LST window = 1 AM LDT D → 1 AM LDT D+1 during DST (standard = midnight→midnight)
- Fetch TWC hourly for the FULL window; 15-min substitutes last hour when within range
- `final_lo = min(cal_lo, min(lst_window_temps))`
- `final_hi = max(cal_hi, max(lst_window_temps))`
- Falls back to cal values for dates >48h out (hourly doesn't reach)
- Hourly and 15-min fetches moved OUTSIDE the date loop (one call per city)

### Why max for highs is correct (verified empirically)
Tested all 19 cities — every correction moved in the right direction:
- Lows: all DOWNWARD corrections (cold front hitting tonight — Chicago −11°F, OKC −20°F, Austin −16°F)
- Highs: all UPWARD corrections (warm rebound on March 13 after front passes — Dallas +11°F, Austin +11°F, OKC +8°F)
The "stale-warm daily, cooler hourly for highs" case doesn't appear in practice. When fronts hit, lows drop but next-day highs RISE. `max` captures both correctly.

### TWC daily self-corrects within ~90 min after midnight
NYC cal_lo was 54°F at midnight EDT; by 1:27 AM EDT it had updated to 36°F. The 18°F staleness gap was real but temporary. The correction still provides value for:
1. Cities that haven't yet updated (Chicago −11°F persisted through the test)
2. Catching warm-rebound highs before the daily updates

## Kalshi API Breaking Changes — March 12, 2026

Kalshi rolled out broad field renames overnight March 11→12. Three separate fixes required.

### 1. Orderbook endpoint: `orderbook` → `orderbook_fp` (fix ~5 AM, commit b866afa)
- `GET /markets/{ticker}/orderbook` now returns `{"orderbook_fp": {"yes_dollars": [...], "no_dollars": [...]}}`
- Old: `{"orderbook": {"yes": [[price_cents, qty], ...], "no": [...]}}`
- `yes_dollars`/`no_dollars` entries are `[price_str, qty_str]` with prices as dollar strings (e.g. `["0.45", "100"]`)
- Fix: Added `_normalize_orderbook()` helper in `trading_utils.py` — tries `orderbook` first, falls back to `orderbook_fp`, converts dollar strings to int cents
- Fill monitor was logging ~40+ API failures per sweep since 3:36 AM

### 2. Trades endpoint: `yes_price`/`no_price` → `yes_price_dollars`/`no_price_dollars` (fix ~5 AM, commit eab0b31)
- `GET /markets/trades` now returns `yes_price_dollars` (str like `"0.0300"`) and `no_price_dollars` instead of int cents
- Also: `count` → `count_fp` (str like `"2.00"`)
- Fix in `paper_trading/fill_monitor.py` `fetch_trades_since()` — supports both formats

### 3. Portfolio orders endpoint + market prices — same pattern (fix 11:15-11:30 AM)

**Orders (`GET /portfolio/orders`):**
- `fill_count` (int) → `fill_count_fp` (str like `"1.00"`)
- `remaining_count` (int) → `remaining_count_fp` (str)
- `yes_price`/`no_price` (int cents) → `yes_price_dollars`/`no_price_dollars` (str)
- Fix: Added `_order_fill_count()`, `_order_remaining_count()`, `_order_price()` helpers in `twc_morning_trader.py` (commit 1035ee9)
- Without fix: `fill_count` = 0 → `_mark_filled` guard fires → every live fill silently skipped → reconcile marks all trades expired

**Market prices (`GET /markets`):**
- `yes_ask`/`no_ask` (int cents) → `yes_ask_dollars`/`no_ask_dollars` (str)
- `parse_market()` in `trading_utils.py` was returning `None` for EVERY market → 0 candidates for all traders
- Fix in `parse_market()`: tries `yes_ask` first, falls back to `yes_ask_dollars` (commit 6372fd1)
- All traders were blind from 9 AM until fix at ~11:28 AM; twc_morning ran at 11:15 and found 0 candidates → force-scan at 11:28 placed 7 orders successfully

### Lesson
Kalshi appears to be migrating away from integer-cent fields to dollar-string fields across the entire API. Watch for the same pattern in any endpoint we haven't audited yet. Key untested: any market list field that uses prices (e.g. `last_price_dollars`, `previous_yes_ask_dollars` visible in raw response).

- The real problem: NBM point forecasts have high uncertainty that isn't captured by XND
- Need to revisit this — inflation as applied may cause more harm than good
- Was in middle of investigating when session compacted

Disambig fires when ASOS reads e.g. 18°C = [64, 65]°F and we bet a between[64-65] YES.

**The model was trained to answer:** "Which °F does this °C reading map to?"
**The question we needed to answer:** "Will the daily max STAY in this bucket?"

These are completely different questions. The daily HIGH can only go up. When current obs = 64°F, the afternoon can still reach 70°F and blow past the bucket. The market already prices in the forecast — when market says YES < 15¢, it means the forecast shows temps rising well above the bucket today.

### Proof from market price buckets (disambig between YES, n=43):

| Market price | n | WR | net_pnl |
|---|---|---|---|
| <15¢ | 16 | 0% | −$176 |
| 15–30¢ | 6 | 0% | −$44 |
| 30–50¢ | 10 | 30% | +$53 |
| >50¢ | 11 | 45.5% | −$44 |

**0 wins on 22 trades when market < 30¢.** The market was right every time. We ignored it.

### Why peak_passed+disambig still fails (25% WR, −$209)
Even when peak_passed says the high is set, disambig between YES loses badly.
Either: (a) peak_passed triggers too early on some days, or (b) "peak passed + current obs in bucket" still doesn't guarantee daily max stays in bucket. The model was seeing a local peak, not the actual daily peak.

### Recommended fix (Ian asked what to do, pending decision)
Options:
1. **Kill disambig between YES entirely** — stop betting that the daily max will stay in a specific °F bucket based on an ambiguous intraday reading
2. **Minimum market price floor**: only trade disambig between YES when market price ≥ 40¢ (market also thinks we're in the bucket). The 11 trades at >50¢ had 45.5% WR — at least directionally correct.
3. Possibly keep disambig on `greater` YES (50% WR, +$33 net on 8 trades — small but positive)
