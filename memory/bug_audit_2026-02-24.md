## 🚨 CRITICAL

### 1. CLI paper trades will never settle — date format bug
**File:** `cli_strategy/trader.py`, `place_paper_order()`  
**Line:** ~430: `date_str = now_utc.strftime("%Y%m%d")`  
**Problem:** Passes `target_date = "20260224"` to `Trade.create()`. The settler looks for
`(city, "2026-02-24")` — hyphen-separated. These never match. All CLI paper trades
are created with an unsettleable date.  
**Fix:** Change to `datetime.now(ET).strftime("%Y-%m-%d")` (or parse from the Kalshi ticker,
which is already done in `parse_market()`).

### 2. `nws_revision_trader.py` — Gaussian σ model (UNLOADED)
**File:** `nws_revision_trader.py`, `p_above()` function  
**Problem:** Uses `P(X > T | X ~ N(mean, σ))` with `σ = max(3.5, abs(revision)/2)`.
Same pattern as the mistake documented twice in LEARNINGS.md. The revision signal
itself might be valid but the probability model is invented.  
**Fix:** Decide with Ian how to compute probability without a distribution —
options: (a) use flat win rate like NWS trader, (b) use ensemble P adjusted by
revision direction, (c) kill the strategy.  
**Schedule:** Was at 0:37 / 6:37 / 12:37 / 18:37 — **currently unloaded**.

## 🔴 HIGH

### 3. `ensemble_trader.py` settle settles ALL models, not just ensemble
**File:** `ensemble_trader.py`, `settle_trades()`  
**Line:** `pending = Trade.get_pending()` — returns ALL models' trades  
**Problem:** The `com.trady.trade-settler` plist (9am daily) runs `ensemble_trader.py settle`.
This settles trades from ensemble, position_updater, nws_obs, cli_confirmed, nws_forecast,
historical_climatology, etc. Portfolio credits are correctly per-model (Trade.settle uses
trade's model), but BotState (legacy global) gets mixed P&L from all of them.  
**Secondary:** `Signal.settle_calibration()` is also called for all models' trades from
the ensemble settler — could contaminate calibration records.  
**Fix:** Change `Trade.get_pending()` → `Trade.get_pending_for_model("ensemble_gfs_ecmwf_icon_gem")`.
Each model should have its own settler, OR trade-settler should be a separate script that
calls `settle_all_models()`.

### 4. `ensemble_trader.py` calls Signal.create with possibly-None trade_id
**File:** `ensemble_trader.py`, `scan_and_trade()`  
**Lines:** ~100-110: `trade_id = Trade.create(...)` then unconditional `Signal.create(trade_id=trade_id, ...)`  
**Problem:** If portfolio is insufficient, `Trade.create()` returns `None`. The Signal is
then created with `trade_id=NULL`, creating an orphaned calibration record.  
**Fix:** Add `if trade_id is None: continue` after `Trade.create()`.  
(position_updater.py already does this correctly.)

### 7. `position_updater.py` current_yes_ask reads wrong orderbook side
**File:** `position_updater.py`, `run_position_updates()`  
**Lines:** ~170-175:
```python
if orderbook and orderbook.get("yes"):  # ← checks "yes" side
    no_bids = sorted(orderbook.get("no", []), ...)  # ← uses "no" side
```
The guard `orderbook.get("yes")` checks the YES side exists, but then reads NO bids.
Should be `orderbook.get("no")`. If `yes` side is empty but `no` side has bids, the
fallback `trade.market_price` is used instead of live data.  
**Fix:** Change guard to `if orderbook and orderbook.get("no"):`.

## 🟢 LOW / COSMETIC

### 11. `compute_exit_pnl` uses ask as exit price (too optimistic)
**File:** `trading_utils.py`, `compute_exit_pnl()`  
**Problem:** Uses current YES ask as the exit price for YES sells. In reality, selling
uses the BID (lower). This overestimates exit proceeds by the bid-ask spread.
Only affects paper P&L estimates for `ensemble_with_updates`.  
**Fix:** Could subtract half the typical spread (≈2-4¢) from exit price.

### 12. `BotState` is a legacy global mixed across all models
**File:** `paper_trading/db.py`, `ensemble_trader.py`  
**Problem:** `BotState` tracks bankroll/total_pnl/winning_trades as a single global.
With 6+ models all sharing it, the numbers are meaningless. `Portfolio` (per-model)
is the right place.  
**Fix:** Eventually stop updating BotState from individual traders. Not urgent.

### 13. `cli_strategy/trader.py` — LOWs are silently skipped in live trading
**File:** `cli_strategy/trader.py`, `find_edge_trades()`  
**Lines:** `if not is_low: ... else: continue`  
**Problem:** CLI trader only uses confirmed HIGH to trade HIGH markets. Low markets
are always skipped. This may miss opportunities where a confirmed high gives strong
info about the low (hot dry day = low can't be too low, etc.).  
**Behavior is intentional** but worth documenting. Could add low trading later.
