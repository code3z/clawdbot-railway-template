### DST boundary findings
- `calendarDayTemperatureMax/Min` uses civil midnight (23-hour day on DST start). Kalshi/NWS uses LST (always 24 hours).
- Disputed hour on DST start (Mar 8→9): midnight EDT to 1AM EDT (UTC 04:00-05:00 Mar 9) — in LST day but not civil day.
- Intraday Evening segment (7PM-1AM) overlaps the disputed hour at its last hour.
- `temperatureMax/Min` (7AM-7PM daytime only) is always within LST — safe floor/ceiling.
- Python ZoneInfo bug: `(aware_dt_a - aware_dt_b).total_seconds()` returns wall-clock 86400 even on 23h DST days. Must use `.astimezone(utc)` before subtracting. Fixed in twc_forecast.py.

### twc_forecast.py — new module written
Location: `polymarket/twc_forecast.py`
Function: `get_twc_forecast(city, target_date, twc_key, cities_cfg) → dict`

Algorithm:
1. Daily API (geocode) → cal_hi, cal_lo, day_hi (7-7), day_lo (7-7)
2. Intraday v1 → identify which 6h segment has the high/low
3. If DST day AND segment is Evening (or Overnight for highs) → fetch hourly 2-day (icaoCode) for full LST window → apply ±1°F intra-hour buffer → also check 15-min if it covers the disputed window
4. final_hi = max(cal_hi, day_hi, disputed_adjustment_if_needed)
5. final_lo = min(cal_lo, day_lo, disputed_adjustment_if_needed)

Test run 2026-03-08 (DST start day): disputed-hour fired for Austin H, Boston L, Seattle HL, Minneapolis H. All other cities returned calendar_day or daytime_7_7 values unchanged.

Rule of thumb: result should equal calendarDayTemperatureMax ~99% of the time. The disputed-hour path only fires on DST transition days when the intraday segment overlaps the disputed window.

## Peak-Passed Model — Precision/Ambiguity Deep Dive (session Mar 8, ~3:30–5 PM ET)

### Key discoveries about ASOS °C→°F precision system

**IEM stores the ORIGINAL °F measurement** (empirically confirmed):
- IEM hourly and 1-min caches contain °F values like 71, 74, 76, 78, 80 that are NOT producible by `round(C*9/5+32)` for any whole °C
- This proves IEM got the original °F from the ASOS system before the °C double-conversion
- ASOS chain: true_temp → convert to °F → round to whole °F → **IEM stores this** → convert back to °C → round → METAR reports °C

**°C bucket structure (critical for gate logic):**
- `asos_display_c_to_f(t_c)` gives [lo_f, hi_f] — the two possible IEM °F values for a given METAR °C reading
- Most buckets contain exactly 2 adjacent °F values (e.g., 22°C → [71, 72])
- IEM 71 and IEM 72 are BOTH from METAR 22°C → they are in the same bucket → a 1°F IEM difference between them is rounding noise, NOT a confirmed temperature change
- A confirmed temperature decline requires different °C buckets

**Exact gate for training/production:**
```python
def f_to_c_bucket(f):
    for c in range(-60, 80):
        lo, hi = asos_display_c_to_f(c)
        if lo <= f <= hi: return c
    return None

confirmed_decline = (f_to_c_bucket(running_max_f) > f_to_c_bucket(current_temp_f))
# confirmed_decline_c = f_to_c_bucket(running_max_f) - f_to_c_bucket(current_temp_f)
```
This replaces the approximate "2°F difference required" heuristic with an exact °C-bucket check.

**5-min ASOS obs are averages of 5 x 1-minute readings:**
- Can be lower than the actual 1-minute peak within the window
- Wethr `new_high`/`new_low` events fire on actual 1-min extremes → authoritative running max
- `running_max_f` for the model must come from `push_extremes` (new_high events) in live mode
- In training: use max IEM reading so far in the day as proxy (Ian confirmed OK)

### Universal rule added to LEARNINGS.md
"Always carry both F values for ambiguous whole-°C ASOS readings. Never collapse to a single point estimate before doing comparisons. The confirmed_signal tier already does this right (max_lower_f/max_upper_f). The probabilistic peak_passed tier needs the same discipline."

## Peak-Passed Model v8b — Data Source Investigation (session Mar 8, ~8-9 PM ET)

### What we learned (the hard way)

**IEM 1-min is broken for training:**
- NCEI phone collection creates large gaps — often entire afternoons missing
- KBOS 2025-07-19: UTC 15:00-23:59 (EDT 11AM-8PM) completely absent → IEM max=74°F vs CLI=87°F (13°F gap)
- ~11% of city-dates have IEM 1-min max < CLI due to these gaps
- DO NOT use IEM 1-min for labels or features

**IEM daily uses DSM not obs:** `max_temp_f` sourced from DSM, 2-3% disagree with CLI by 1-2°F. Use CLI cache as ground truth.

**NWS API:** only 2 days of obs history. Useless for training.

**IEM 5-min temps:** intentionally set to MISSING since 2016 (F→C→F rounding issue). Can't use.

**Synoptic Data is the answer:** Returns actual HF-METAR whole-°C (exactly what wethr sends). Token: `b77ae1e010e344d8863bf53aeb8e5ac1`. Coverage 2025+ on our plan. No gaps. Confirmed matches CLI.

**Label:** `peak_is_set = (5min_running_max_c >= 5min_daily_max_c)` — same source for both sides. NOT CLI, because CLI uses 1-min and we can't observe those spikes from 5-min data.
