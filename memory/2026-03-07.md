## Morning Execution Strategy Backtest (completed ~2 AM ET)

### Goal
Design and validate a limit-order execution strategy for `twc_morning_trader` that maximizes fill-price savings while guaranteeing 100% fill rate by 9 PM ET.

### Dataset

- **Stored at**: `/tmp/backtest_combined.json`

- Market metadata: `/tmp/kalshi_markets_v2.json` (24,470 settled markets across 20 series)
- 151 NO bets (B-suffix / between markets), 447 YES bets (T-suffix / threshold markets)
- Vol filter: ≥100 contracts

### Strategy Tested — S15 IS THE WINNER

**S15: Rolling drop check + S3 escalation**

Logic (per hour, starting at h=1):
1. If `ask_close < open_ask - 3`: take market immediately at `ask_close`
2. Otherwise follow S3 schedule:
   - h ≤ 4: limit = `open_ask - 5`
   - h ≤ 8: limit = `open_ask - 2`
   - h ≤ 10: limit = `prev_close` (sync to market)
   - h > 10: limit = `prev_close + 1` (aggressive)
3. Cap all fills at `open_ask + 1` (never pay more than 1¢ above open)
4. Fallback: take market at whatever price at end of window

**Results on 598 markets:**
- Saves: **+2.45¢ avg** vs buying at open (100% fill rate)
- S3 was previous best: +1.79¢
- Worse than open: 23% of markets (bounded by +1¢ cap)
- Median saving: +2¢, p90: +5-6¢
- Beats S3 in 22% of markets, same 69%, worse only 9%
- Consistent across bet types: NO saves +2.38¢, YES saves +2.48¢

**Why S15 beats S3:** S3's -5¢ limit only fills when price *touches* open-5¢. S15 also captures sustained dips (e.g. ask drifts to open-4¢ and holds there) by checking ask_close each hour.

### Key Findings
- **Cap is non-negotiable**: S10 (no cap) saved only +0.7¢ — removing cap halves savings
- **Bet-type specialization doesn't help**: Same strategy works for both YES and NO bets
- **Checking frequency**: Hourly `ask_low` already captures sub-hourly dips; going to 1-min doesn't change fill prices, just timing
- **Empty book**: Always a posted quote (spread 1-2¢), but vol_h1=0 is real for thin markets; fallback handles it
- **S15 is ready to implement** in `twc_morning_trader.py`

### CRITICAL FINDING: v2 Table Was Entirely Wrong

v2 (OM-vs-OM) said p_peak_set = 98% when om_bin = -3_or_less.
v3 (IEM-vs-CLI) says p_peak_set = only **41%** for the same bin.

**Why the huge gap?** `om_bin = -3_or_less` conflates two very different situations:
1. Late afternoon (H=15–20): running_max is well above OM's ceiling → peak almost certainly set (~90%+)
2. Early (H=11–12): OM forecast was too cold (e.g. said 60°F max, we're already at 63°F at 11 AM) → temperature still rising, CLI will settle at 68°F → peak NOT set

The mixture across hours 11–20 averages to ~41% — which is nearly useless as a signal.

**Root problem: `om_remaining_upside_f` alone is NOT a good primary feature for highs.**
- For lows: OM is well-calibrated for overnight minimums → downside table works
- For highs: when OM is wrong (we're above its forecast), temps KEEP going up → table degrades
- Better primary features for highs: `local_hour` + `hours_past_predicted_peak` + `obs_decline_f`

### Peak Model v4 — sklearn GBC with 15 features (COMPLETE)

**Files created:**
- `peak_trough_data/build_peak_model_v4.py` — feature build + training script
- `peak_trough_data/peak_model_features_v4.csv` — 81,140 rows (same corpus as v3, augmented features)
- `peak_trough_data/peak_sklearn_v4.pkl` — GBC (300 est, depth=5, lr=0.04) + isotonic calibration
- `peak_trough_data/peak_model_v4_analysis.md` — Brier + calibration analysis

**New features (v4 adds 9 on top of v3's 6):**
- `obs_peak_hour`: IEM local hour when running_max first achieved (not OM predicted hour)
- `hours_since_obs_peak`: H − obs_peak_hour (sustained decline duration per IEM obs)
- `obs_1h_delta`: IEM temp[H] − temp[H-1] × 9/5 (recent trend in °F; negative = declining)
- `obs_2h_delta`: IEM temp[H] − temp[H-2] × 9/5
- `om_h0_vs_max`: OM forecast[H] − running_max_f (how far OM is below peak)
- `om_next3h_decline`: OM[H] − OM[H+3] (OM forecast slope next 3 hours)
- `om_already_declining`: 1 if OM[H] < OM[H−1]
- `hpp_times_decline`: hours_past_predicted_peak × obs_decline_f (interaction term)
- `obs_since_peak_x_om`: hours_since_obs_peak × max(0, −om_remaining_upside_f) ← top feature (54% importance)

**Results (train Jan–Oct 2025, test Nov 2025–Feb 2026):**
- Brier: v3=0.0788 → v4=**0.0692** (12% improvement)
- p≥0.70 fires: 2,563 → **2,916**; win rate: 0.948 → **0.958**
- Calibration: well-calibrated at p≥0.85; under-confident at 0.30–0.85 (actual ~1.2–1.4x model output)
- Simulation at p≥0.40, ask=50¢: n=3,472, win=0.927, avg pnl=+0.427/trade

**Key insight: high-p fires are extreme cases (dec=20+°F, hours_since=15+h)**
- The slam-dunk cases (Jan cold fronts etc.) naturally hit p→1.0
- Practical afternoon scenarios (dec=6-8°F, hours_since=3-4h) output p=0.20-0.40, but calibration shows actual win rate ~30-56% there
- Root cause: `obs_since_peak_x_om` interaction dominates (54% importance), creates near-binary split between extreme/non-extreme cases

**`obs_since_peak_x_om` is the breakthrough feature:**
- For a Jan cold front day: hours_since=15, om_upside=24 → x_om=360 → p→1.0
- For a normal afternoon: hours_since=3, om_upside=4 → x_om=12 → p≈0.20
- Correctly models: "how long has the IEM been in decline AND is OM confirming no further rise?"
