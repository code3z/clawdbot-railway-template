Evidence: KMDW had prev_value_c=3, prev_value_f=37 then value_c=3, value_f=38 —

(e.g. NWS API returning whole °C with no °F field).

## CORRECTION #2: Wethr new_high value_f is NOT exact (12:47 PM)

Overriding earlier "value_f is exact" conclusion — Ian was right that it's rounded.

Evidence:
- obs_cache temp_f_display=37.4 (raw °C×1.8+32) for KMDW 3°C readings
- new_high value_f=38 for same 3°C reading
- 37.4 rounds to 37 (standard round), but ceil(37.4)=38 → wethr uses CEILING for daily high value_f
- So wethr's new_high value_f can differ from NWS official settlement by 1°F

**Do NOT use new_high value_f as exact °F. It's ceil(°C conversion), not NWS settlement.**

Correct model for fetch_obs_from_push:
- Read obs_cache rows for station+date
- lowest_probable_f / highest_probable_f already stored correctly via asos_display_c_to_f
- daily min_lower_f = MAX(lowest_probable_f) across today's obs
- daily max_upper_f = MAX(highest_probable_f) across today's obs
- Augment with push_extremes (cli/dsm events are exact — use those directly)

obs_cache schema reminder: temp_f_display (decimal, raw conversion), lowest_probable_f (int), highest_probable_f (int) — use the latter two.

## 🚨 Critical Low-Market Bug Fixed (commit ff3f8e5, 1:59 PM)

**The bug**: confirmed_low_signal had inverted logic — was firing "greater YES" when
running min > floor, claiming "low is definitely above floor." WRONG. The daily low
can only go DOWN. Current min of 55°F does NOT confirm low > 54°F — it could drop
to 45°F tonight.

**The rule (burned in)**:
| | High (can only ↑) | Low (can only ↓) |
|---|---|---|
| Confirmed ABOVE threshold | ✅ monotonic | ❌ needs end-of-day |
| Confirmed BELOW threshold | ❌ needs end-of-day | ✅ monotonic |

**Valid confirmed-low signals (intraday, monotonic)**:
- greater NO:       min_upper_f ≤ floor → low definitely ≤ floor → NO wins ✓
- between NO (low): min_upper_f ≤ floor → below range → NO wins ✓
- less YES:         min_upper_f ≤ cap   → low definitely ≤ cap → YES wins ✓

**Invalid (removed)**:
- greater YES — current min > floor, but low could still drop tonight
- less NO — current min > cap, but low could still drop below cap
- between NO (above range) — same issue

First wrong signal caught and killed before any live trade: LAX T54 YES was bogus.

## Trough Model Calibration Complete (2:53 PM)

Script: `calibrate_trough_model.py` | Output: `trough_data/trough_model.json`
2,190 records × 6 decision hours = 13,140 simulated observations.

**THE KEY FINDING: consec_rising is a near-perfect gate**

Once 3+ consecutive 5-min ASOS readings are above the current observed daily min,
P(trough already passed) = ~100% regardless of gap or city. No per-city split needed.

**Decision rule (at 8am local, all cities):**
| consec_rising | fc_gap       | P(trough passed) | Trade?          |
|---------------|--------------|------------------|-----------------|
| ≥ 3           | any < 0      | ~100%            | Yes, conf=0.95  |
| 0-2           | ≤ -8°F       | 99.5%            | Yes, conf=0.95  |
| 0-2           | -8 to -6°F   | 94%              | Yes, conf=0.92  |
| 0-2           | -6 to -4°F   | 89%              | Yes, conf=0.87  |

## Wethr Push Client — 409 Incident (7:59–8:39 PM ET)

### What happened:
1. Heartbeat monitor (10s check loop) fired after 450s with no "heartbeat" event type
2. Monitor called `resp.close()` and tried reconnecting
3. But monitor kept firing every 10s BEFORE old connection died → multiple close+reopen cycles
4. Server (1-connection limit) saw zombie connections → started returning HTTP 409

2. `HEARTBEAT_TIMEOUT` raised 450s → 900s (now truly a last-resort safety net, not routine)
3. HTTP 409 backoff: 300s wait (5 min) instead of short exponential — prevents hammering server when it still sees old connection

- 300s backoff means it retries quietly every 5 min — will self-recover

### Key lesson:
When SSE heartbeat monitor fires and closes the stream, the server-side connection may not die immediately. Multiple rapid reconnect attempts after a forced-close = connection pile-up = 409. Always use a long backoff (300s+) on 409 responses.
