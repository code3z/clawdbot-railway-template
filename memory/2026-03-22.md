### Key insight: six_hour_high mechanism
- new_high events fire throughout the day as temperatures rise (well before synoptic boundary)

- Normal pattern: six_high_f = cur_high + 0 or +1 (disambiguation)
- SFO March 18 anomaly: +4°F → true spike in the METAR 1SNNN field

## obs_peak_model_trader (v9) — Deep Diagnosis (Mar 22 ~00:00–09:30 UTC)

### What we established
- Trader IS running v9 (not v8b). The `peak_passed_v8b` tags are from before March 16. Header comment just wasn't updated.
- TWC_PEAK_MODEL evaluator also runs inside the same trader as a separate signal.

### Root cause 1 — Dedup failure (351 rejected trades)
- `Trade.exists_for_model` only checks `status IN ('pending','pending_fill')`
- FOK orders that get rejected → `status='rejected'` → not blocked → same market retried every 5-min push event
- Worst ticker: `KXHIGHTSFO-26MAR18-T79` appeared 63 times
- The rejections themselves cost no money (FOK = clean reject, no exposure)
- **Real harm**: wasted API calls + DB clutter + misleading stats

### Root cause 2 — Time gate not applied to push events
- The scan has `if not (10.0 <= local_h <= 21.5): continue` — push events don't
- `on_peak_push_event` calls `_evaluate_city` directly with zero time filtering
- Wethr push events at 8am trigger v9 before the daily peak has formed
- Gate trader doesn't have this problem — its triple-condition gate (30+ min since obs peak, ≥1°C decline, 60+ min past OM peak) naturally prevents early firing

### Root cause 3 — `record_rejected` hardcodes our_p=0.0
- `Trade.record_rejected` stores `our_probability=0.0` with comment "# not in TradeRequest" — it IS in TradeRequest
- Cosmetic bug: actual trading computation was correct, only DB record is misleading
- All 351 rejected records show `our_p=0.00`

### Root cause 4 — Model applied to wrong situation (`less/YES` with high p2plus)
- **The actual cause of losses**: the model is being applied to `less/YES` markets where p2plus is still high (0.50–0.93)
- When p2plus > 0.20: model is saying "lots of upside likely" but we compute p0=5-43% and trade anyway if `p0 - market_price ≥ MIN_EDGE`
- **All 4 wins**: p2plus ≤ 0.12 (model genuinely confident peak is in)
- **Almost all losses**: p2plus 0.28–0.93 (model uncertain, still trading)
- Fix: gate `less/YES` on `p2plus < 0.20` before computing our_p → don't trade when model is saying upside is likely

### Training data — clarifications
- Training uses hourly sampling (`for hour_t in range(24)`) NOT push events — no sampling bias
- class 0 rate at h=8 is 12.84% (overall). Spring: 14.1%. Winter: 18.8%. Not dramatically different.
- The model IS correctly calibrated for local_hour. The p2plus signal is the discriminator.

### The correct model interpretation for `less/YES`
- When p2plus is low (< 0.20): model confident the high is in → good signal → trade
- When p2plus is high: model is uncertain/wrong scenario — the question "will high stay below cap?" can't be answered by bucket-level class 0/1 probabilities alone when there's still 80%+ chance of further upside
