## P_TABLE Calibration — Regression Fixed-Point Analysis (Mar 22 ~17:00–17:30 UTC)

### What we tried
Attempted Option B: fit logistic regression on all training rows → solve P* = regression(delta, P*−MIN_EDGE, type) numerically via brentq.

### First attempt failure
- Included yes/between rows (delta=0, WR=33%) in training → `is_no` coefficient learned wrong sign → all NO cells snapped to 0.090 ("never trade"), all YES cells to 0.999. Garbage.

### Second attempt (correct: exclude yes/between)
- Training set: 2772 rows (excl yes/between — we never trade them, they distort model)
- Log loss: 0.3392 vs baseline 0.3855 — good fit
- Coefficients: delta×noon (+0.9942) dominant, noon (+0.5760), is_no (-0.1241), delta×no (-0.1838)

### Fixed-point result
- **46/64 cells → "never-trade" (P*≈0.09)** — ALL NO cells
- **14/64 cells → clean P*** — ALL YES cells (yes/greater and yes/less)
- This is NOT a model bug. The "never-trade" result for NO is CORRECT given MIN_EDGE=0.08.

### Key insight: NO trades have thin edge
Direct edge scan result:
- NO/between/below delta=1: max edge = **-1.1%** (actually negative!)
- NO/between/below delta=2: max edge = **+2.2%** at noon=0.76
- NO/between/below delta=3: max edge = **+4.4%** at noon=0.76
- NO/between/below delta=4: max edge = **+6.1%** at noon=0.77
- NO/between/below delta=5: max edge = **+7.4%** at noon=0.77

None clear MIN_EDGE=0.08 (8%) except barely at delta=5.

**YES/less trades**: max edge = **24–30%** consistently across all deltas, peaking at noon=0.51–0.54. Huge genuine alpha.

### Why this confirms our trading results
- NO trades in practice: 68% WR, -$100 net — market is well-calibrated for NO, thin edge
- YES trades in practice: 100% WR on legitimate trades, +$126 net — market systematically underprices YES
- Our edge comes from YES trades, not NO trades. P_TABLE has been overstating NO edge.

### Practical implication
- To find NO fixed-points, would need MIN_EDGE ≈ 0.03–0.04 (3-4%)
- YES fixed-points exist cleanly: yes/less and yes/greater converge at P*=0.94–0.99

Root cause of twc_pace 429s: `twc_portfolio_trader._build_candidates()` was calling `get_best_ask()` → `fetch_orderbook()` (GET /markets/{ticker}/orderbook) for EVERY candidate market × 2 (yes + no) EVERY 10-second loop. ### Fixes applied

4. **"Never Make Uncached API Calls Inside Hot Loops" rule rewritten**: Ian pointed out this rule was wrong/too broad. Real concerns are (a) calls/sec vs rate limit and (b) data freshness. ## cli_confirmed — Replay Dedup + Stale wethr_high Fix (Mar 22 ~20:00–21:09 UTC)

### Bug 3: Server event ID resets cause observation replay

The wethr SSE server resets its event ID space periodically (observed: 9677 at 00:25 UTC → 5623 at 15:19 UTC). When the daemon reconnects with a stale ID from the previous server session, the server replays old events. Previously these replayed observations (including six_hour_high) passed through without filtering.

**Fix: observation replay dedup** (commit bdf0a2e)
- On startup, load `MAX(obs_time)` per station from `obs_cache` into `_latest_obs_ts` dict
- In `_handle_event`, for all station events with `observation_time_utc`: if `obs_time <= _latest_obs_ts[station]` → silently skip (stale replay)
- Key: uses obs_time (from actual measurement), NOT server event ID — immune to server ID resets
- `_latest_obs_ts` updated in-memory as new obs are written (via `_write_obs_cache`)
- Startup log: `"📋 Loaded latest obs timestamps for 20 stations (replay dedup active)"`
- Events without `observation_time_utc` (meta events: connected, heartbeat, cli, dsm) are never filtered

**Note on `new_high` events**: They DO carry `observation_time_utc` (format: `"YYYY-MM-DD HH:MM:SS"`, space-separated, UTC). String comparison is correct and sortable. They ARE filtered by the dedup check. `_latest_obs_ts` is only updated by `_write_obs_cache` (observation events), not new_high events — but since observation events come in continuously at 5-min intervals, the latest obs time is always current and new_high events from replay (older timestamp) are correctly filtered.

### Bug 4: wethr_high.nws persists across LST midnight — root cause of ATL 84°F

**This is the ACTUAL root cause** (not the DSM dispatch or six_hour_high replay):

Wethr's `wethr_high.nws` field is a rolling station high that does NOT reset at LST midnight. Early morning observations on day D (e.g., 05:10 UTC = 00:10 LST for ET) carry `wethr_high.nws.value_f = 84` from March 21's peak. `_lst_date("KATL")` at 05:10 UTC returns "2026-03-22", so the observation handler was writing 84 to `push_extremes["KATL"]["2026-03-22"]`.

**Evidence from wethr_push.jsonl:**
```
obs at 2026-03-22T00:02:27 UTC:
  observation_time_utc: "2026-03-22 00:00:00"
  local_date: "2026-03-21"       ← wethr's own date
  wethr_high.nws.value_f: 84    ← March 21's peak
  wethr_high.nws.time_utc: "2026-03-21 21:40:00"  ← when the high was observed
  high_valid_through_utc: "2026-03-21 23:52:00"   ← stored but NEVER used as guard
```

`high_valid_through_utc` was stored in the DB but the observation handler never used it to decide whether to write the high. The function wrote `wethr_high.nws.value_f` unconditionally whenever the value was non-null and higher than what was stored.

**Fix: timestamp guard before writing wethr_high** (commit e7708df)
In the observation handler, compute today's LST midnight in UTC:
```python
city_   = STATION_TO_CITY.get(station)
tz_off  = _LST_OFFSETS.get(CITIES[city_]["tz"], -5) if city_ else -5
lst_midnight_utc = f"{date_str} {abs(tz_off):02d}:00:00"
# e.g., "2026-03-22 05:00:00" for ET (UTC-5)
```
Only write `wethr_high.nws.value_f` to today's push_extremes if `wh["time_utc"] >= lst_midnight_utc` — i.e., the high was actually observed within today's LST calendar day. Same guard applied to low.

**Verification:**
- March 21 21:40 UTC < 2026-03-22 05:00:00 UTC → SKIP writing 84 to March 22 ✅
- March 22 14:28 UTC (first real new_high) >= 05:00 UTC → write ✅

**Chain of confusion:** DSM and six_hour_high appeared to be the trigger because they "confirmed running max = 84". But they were confirming a value already wrongly written by the observation handler earlier in the day. The observation handler bug was writing 84 at every 5-minute obs cycle from 05:00 UTC onwards. Six_hour_high at 11:54 UTC just confirmed what was already there.
