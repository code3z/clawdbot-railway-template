### wethr push temperature precision — key lesson

**wethr sends WHOLE-°C values, not raw °F.** `nws_high_f` in `push_extremes` is the °F value from the push event but derived from a whole-°C reading. This means:
- The [lo_f, hi_f] ambiguity from `asos_display_c_to_f(value_c)` is real
- When `nws_high_f == bucket_hi_f`: CONFIRMED EXACT at hi_f (ASOS measured exactly that °F)
- When `nws_high_f == bucket_lo_f`: we know it's lo_f currently, but temp could advance to hi_f later in the day → conservative approach: use bucket_hi_f for trade decisions

Evidence from JSONL: KLAX fired two separate new_high events on same day: value_f=69 (lo of 21°C=[69,70]) then value_f=70 (hi of 21°C). The push stream fires on EVERY whole-°F increase, not just °C bucket changes.

**Current code behavior: CORRECT.** Uses bucket_hi_f always — conservative when nws_high_f=lo_f (temp could still reach hi_f), exact when nws_high_f=hi_f.

**My mistake:** Wrote a "fix" to use raw nws_high_f directly as hi_f, then reverted after Ian pointed out the memory note "wethr push (5-min whole-°C, bucket ambiguity)." The revert was correct.

### Future enhancement identified (not yet built)
When CLI report, 6-hr METAR, or DSM confirms the period max was only lo_f, we KNOW the temp never reached hi_f. This lets us trade with gap = cap - lo_f (larger) instead of cap - hi_f (smaller). Opens trades that currently look too close to cap.

Sources that give exact °F (no bucket ambiguity):
- CLI report (exact whole-°F daily high)
- 6-hr synoptic METAR max (exact, 0.1°C → single °F via `metar_tenths_c_to_f`)
- DSM (exact settlement value)

wethr push already fires `six_hour_high` events — plumbing these into push_extremes with an `exact` flag (not just lo_f==hi_f single-value buckets) is the implementation path.

## Late Afternoon Session — forecast_model portfolio fix + March 24 trades (Mar 23 ~17:15-17:37 UTC)

### Portfolio balance bug root cause
`_repair_portfolios()` in `paper_trading/db.py` runs at every `init_db()` call and recomputes:
`balance = starting_balance + realized_pnl - pending_exposure`

29 pending trades at avg $63 = $1,785 committed → balance = $500 - $1,785 = -$1,285.
Direct SQL resets to $500 are immediately overwritten on next `init_db()`. The fix: change `starting_balance` to $5,000 for both `forecast_model_v1` and `forecast_model_v2`. Done.

Also note: `Trade.open_pending()` (used by forecast_model traders) sets `status='pending_fill'` and **does NOT deduct from portfolio** at creation. Fill monitor later calls `Trade.fill()` which changes status to `pending` but also does NOT touch portfolio balance. So `_repair_portfolios` is the only thing that actually sets the balance, using the pending exposure calc above.
