## Complex Model Built (Seattle focus)
- `polymarket/weather_complex.py` — multi-model ensemble with probabilistic bucket pricing
- Target: Seattle (KXHIGHTSEA) — 105k vol, 12 markets/day, 3.1¢ spreads
- Uses 7 Open-Meteo models (GFS, ICON, GEM, MeteoFrance, JMA, UKMO, ECMWF) + NWS forecast
- Builds normal distribution from weighted model consensus → prices each bucket → finds edge vs market
- Model weights start equal (NWS gets 1.5x bonus), auto-update based on historical accuracy
- First scan found 2 trades: BUY NO on 52-53° (20.5% edge, NWS anchoring), BUY YES on 48-49° (16.9% edge, market ignoring model cluster)
- Runs at 3pm, 3:30pm, 6pm, 10pm, 6am ET (matching forecast update cycles)

## Critical Bugs Found & Fixed (morning session with Ian)

### Bug 1: UTC timezone in observation queries
- `nws_observations()` queried from midnight UTC (= 7pm ET previous day)
- Result: yesterday's warm evening temps included as "today's" readings
- DC frontrunner saw 44.6°F (from 8-9pm ET Feb 21) and placed 3 "LOCKED" NO trades on dead buckets that weren't actually dead
- **Fix**: Each city now has its own IANA timezone. Observations query local midnight → local midnight.

### Bug 2: Kalshi NO trade pricing inverted
- `place_order()` always passes `price_cents` as `yes_price` to Kalshi API
- For NO trades, `price_cents` was `no_ask` — but Kalshi API always takes `yes_price`
- Buying NO at `no_ask=34` sent `yes_price=34`, which Kalshi interprets as NO costing 66¢
- **Result**: Miami overpaid 32¢/contract, NYC overpaid 14¢/contract. Most NO orders sat resting because the inverted price was too cheap for anyone to take.
- **Fix**: `execute_signal()` now converts: `yes_price = 100 - no_ask` when side="no"

### Bug 3: Resolver crash
- `resolve_trades()` referenced `now` variable that was renamed to `now_utc`
- Would crash on every resolution attempt. No trades ever resolved through v2.

### Bug 4: Low temp forecast date assignment
- NWS "Tonight Feb 22" overnight low actually occurs early morning Feb 23
- Was assigning to Feb 22 — wrong calendar day for Kalshi resolution
- **Fix**: overnight low period assigned to next calendar day

### Bug 5: Traded tickers never expire
- `load_traded_tickers()` loaded all historical tickers — unbounded growth
- **Fix**: 7-day rolling window

### Lesson
- **Kalshi YES_PRICE gotcha strikes again** — this is the THIRD time (see MEMORY.md). Must be burned into muscle memory: Kalshi API ALWAYS takes yes_price. To buy NO at X¢, send yes_price = 100 - X.
- **Always test with real API responses before going live** — a dry-run that doesn't hit the API won't catch pricing bugs
- **Timezone bugs are silent killers** — code appeared to work, even placed "successful" trades, but on wrong data
